# AGENT.md ‚Äî Flutter App Template

A **purely offline, on-device** application built with **Flutter + Dart**. No backend, no accounts, no network calls, no cloud storage. All data lives in a local SQLite database via **drift** on the user's device.

**Key references:**
- Product requirements: [`docs/PRD.md`](docs/PRD.md)
- Visual identity: [`docs/brand-kit.md`](docs/brand-kit.md)

---

## 1) Core Principles

1. **Offline-first & privacy** ‚Äî Zero network requests. Zero accounts. Zero analytics. The app MUST function with 100% feature parity in airplane mode. Never request unnecessary permissions.
2. **Type safety** ‚Äî Dart sound null safety everywhere. Drift constraints validate database I/O. No `dynamic` types except where required by framework APIs.
3. **Small, reviewable changes** ‚Äî Prefer incremental PRs over large rewrites. Each commit MUST leave the app in a buildable state.
4. **Test-first for core logic** ‚Äî Red-Green-Refactor for repository CRUD, validation, computed fields, and aggregation queries. Widget tests for UI components.
5. **Simplicity & YAGNI** ‚Äî Only build what the PRD defines for v1. Three similar lines are better than a premature abstraction.

## 4) Data Layer

### Drift Guidelines

- Define tables as Dart classes extending `Table` in `app_database.dart`
- Use `text()` columns with ISO 8601 strings for datetimes (store local time)
- Use `integer()` columns with `.check()` constraints for bounded values
- Use `text().nullable()()` for optional fields
- Define a `primaryKey` override on every table
- Add indexes on columns frequently used in queries or sorting

### Repository Pattern

Each entity should have a repository class that:
- **Validates** all inputs before writing to the database
- **Computes** derived fields (e.g. durations, derived dates) on save
- **Generates** UUIDs for new records
- Exposes `create`, `list` (with optional filters, pagination), `update`, and `delete` methods
- Uses drift `Companion` objects for inserts and updates

### Drift API Notes

- Run `dart run build_runner build` after any change to table definitions or queries
- Use `NativeDatabase.memory()` for in-memory testing (no mocks needed)
- Schema migrations: increment `schemaVersion` in the database class, add migration steps in `MigrationStrategy`
- Use `Companion` objects for inserts and updates

## 5) UI/UX Standards

### Component Rules

- Use **`InkWell`** or **`GestureDetector`** for tap interactions ‚Äî never raw `Listener`
- Use **`ThemeData`** and the app's theme extension for all colours and text styles ‚Äî never hardcode values
- Use **`SizedBox`** for spacing between elements
- Trailing commas on all widget constructor arguments for consistent formatting

### Screen States

Every screen MUST handle these 4 states, implemented via Riverpod's `AsyncValue.when()`:

1. **Loading** ‚Äî skeleton or `CircularProgressIndicator`, never a blank screen
2. **Empty** ‚Äî helpful message + CTA to create the first entry
3. **Error** ‚Äî user-friendly message with retry action
4. **Content** ‚Äî the normal data-present state

### List Performance

- Use `ListView.builder` for any list that could grow beyond a few items
- Use `const` constructors on list item widgets where possible
- Extract list items into their own widget class to enable const construction

### Accessibility

- All interactive controls MUST have `Semantics` labels
- Minimum touch target: 48x48dp (Material Design) / 44x44px (Apple HIG)
- TalkBack (Android) and VoiceOver (iOS) verified for key journeys
- Follow contrast rules from the brand kit
- Colour MUST NOT be the sole indicator of meaning

### Brand & Visual Identity

Follow `docs/brand-kit.md` as the single source of truth for:
- Theme mode (dark/light)
- Primary and accent colours
- Typography (font families, weights, sizes)
- Icon style and sizing
- Spacing grid

## 6) Testing Strategy

### Configuration

- **flutter_test** (built-in) for unit and widget tests
- Drift **in-memory database** (`NativeDatabase.memory()`) for repository/service tests ‚Äî no mocking SQLite
- **mocktail** for mocking dependencies (e.g. repositories in widget tests)
- Riverpod testing: `ProviderContainer` for unit tests, `ProviderScope.overrides` for widget tests

### Test Organisation

| Type | Location | What to test |
|---|---|---|
| Unit | `test/repositories/` | Repository CRUD, constraint enforcement, computed fields |
| Unit | `test/services/` | Business logic, aggregation queries, edge cases |
| Widget | `test/screens/` | Each screen renders correctly in all 4 states |
| Widget | `test/widgets/` | Shared widget behaviour and appearance |
| Integration | `integration_test/` | Key user journeys on-device |

### What MUST Be Tested

- All constraint/validation rules defined in the PRD
- Computed/derived field calculations
- Edge cases: empty data, boundary values, invalid inputs
- Aggregation queries with missing data / no data
- Schema migration preserves existing entries

### Run Commands

```bash
# All tests
flutter test

# Specific test file
flutter test test/repositories/foo_repository_test.dart

# With coverage
flutter test --coverage

# Integration tests (requires running emulator/simulator)
flutter test integration_test/
```

## 7) Known Gotchas

Hard-won lessons ‚Äî consult this before debugging.

| Topic | Gotcha | Fix |
|---|---|---|
| build_runner | Generated files (`*.g.dart`) are stale after drift/Riverpod/freezed changes | Re-run `dart run build_runner build --delete-conflicting-outputs` |
| Generated files | Editing `*.g.dart` files manually causes them to be overwritten on next build | Never edit `*.g.dart` ‚Äî modify the source file and regenerate |
| Drift migrations | Schema changes without migration steps cause runtime crashes | Increment `schemaVersion` and add migration logic in `MigrationStrategy` |
| Custom fonts | Fonts not rendering despite being in `assets/fonts/` | Must register each font family + weight in `pubspec.yaml` under `flutter: fonts:` |
| fl_chart sizing | Charts render as zero-size or overflow without explicit constraints | Wrap fl_chart widgets in `SizedBox` or `AspectRatio` with explicit dimensions |
| AsyncValue.when | Forgetting to handle all 3 cases (data, loading, error) causes compile errors | Always provide all 3 callbacks: `data:`, `loading:`, `error:` |
| DateTime storage | Mixing UTC and local datetimes causes off-by-one date issues | Store local ISO 8601 strings; derive date keys from local time |
| intl locale | `DateFormat` defaults to en-US if `initializeDateFormatting` not called | Call `initializeDateFormatting()` in `main()` before `runApp()` |
| Riverpod codegen | `@riverpod` annotation changes don't take effect immediately | Re-run build_runner after adding/modifying `@riverpod` annotated functions |
| go_router redirects | Redirect loops when guard logic is wrong | Test router configuration in isolation; avoid circular redirect conditions |
| Drift companions | Using the wrong companion type for insert vs update | Use `FooCompanion.insert(...)` for new rows, `FooCompanion(...)` for updates |
| Platform differences | fl_chart rendering differences on iOS vs Android | Test charts on both platforms early; use explicit colours for chart data |
| pubspec.yaml | Incorrect indentation silently breaks asset/font registration | Validate YAML indentation; run `flutter pub get` and check for warnings |

## 8) Agent Workflow & Conventions

### How to Operate

When you receive a task:

1. **Restate the goal** in 1‚Äì2 sentences
2. **Read existing code** ‚Äî understand patterns before modifying
3. **Make the smallest coherent change** that solves the problem
4. **Update or introduce:** types, drift schemas, tests, docs (if behaviour changes)
5. **Verify:** no network calls added, all 4 screen states handled, Semantics labels present
6. **Summarise:** files changed, how to test

### Coding Conventions

- Dart **sound null safety** ‚Äî no `dynamic` types, no `!` operator unless provably safe
- File names: `snake_case.dart`
- Class/enum names: `UpperCamelCase`; variables/functions: `lowerCamelCase`
- Trailing commas on all widget constructor arguments
- Prefer `final` over `var`; use `const` constructors wherever possible
- No duplicate business logic ‚Äî extract to `lib/utils/` or `lib/services/`
- Functions should be small and pure where possible
- No "magic strings" ‚Äî centralise route paths, constants
- Hoist expensive constructors (`DateFormat`, `NumberFormat`, etc.) to top-level or static fields

### What NOT to Do

- **Do not** add any backend, BaaS, or cloud service unless the PRD requires it
- **Do not** add authentication or session management unless the PRD requires it
- **Do not** add any library that makes network requests
- **Do not** add analytics SDKs that transmit data off-device
- **Do not** use Bloc, GetX, or Provider (we use Riverpod)
- **Do not** use `setState` in screens ‚Äî use Riverpod providers for all state
- **Do not** manually edit generated `*.g.dart` files
- **Do not** add screens or navigation paths beyond what is defined in the PRD
- **Do not** refactor unrelated code "for cleanliness" during feature work
- **Do not** introduce new libraries without clear justification

## 10) Code Generation Workflow

Drift, Riverpod (with `riverpod_generator`), and freezed all rely on `build_runner` to produce `*.g.dart` files. This is a fundamental workflow concern.

### When to Run

Re-run code generation after any of the following changes:
- Adding or modifying drift **table definitions** or **queries**
- Adding or modifying **`@riverpod`** annotated providers
- Adding or modifying **`@freezed`** annotated classes
- Changing drift **schema version** or migration logic

### Commands

```bash
# One-off build (recommended before commit/test)
dart run build_runner build --delete-conflicting-outputs

# Watch mode (continuous rebuilds during development)
dart run build_runner watch --delete-conflicting-outputs
```

### Rules

1. **Never** edit `*.g.dart` files ‚Äî they will be overwritten on next build
2. **Commit** generated files to version control (they are needed for CI and other developers)
3. **Run build_runner** before running tests or committing ‚Äî stale generated files cause confusing failures
4. Use `--delete-conflicting-outputs` to avoid stale file conflicts
